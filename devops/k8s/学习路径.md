学习路径

```
├───k8s-learn
│   ├───01-architecture (架构：Control Plane, kubelet, etcd)
│   ├───02-objects (核心对象：Pod, Deployment, Service, Namespace)
│   ├───03-storage (存储：PV, PVC, StorageClass)
│   ├───04-config (配置：ConfigMap, Secret)
│   ├───05-networking (网络：CNI, ClusterIP, NodePort, Ingress)
│   ├───06-scheduling (调度：Affinity, Taints, Tolerations)
│   └───07-helm (包管理：Helm Chart 编写)
```

现在这个项目的生产形态，用 `docker compose + 单台 GCP free tier VM` 是合理的。

1GB 内存的机器上强上 k8s，工程上没有收益，反而会让系统更脆弱。为了求职而在本地单独做一个 `k8s-lab` 分支，用同一个项目去演示容器编排、部署、配置管理、服务发现、持久化、健康检查、滚动更新，这个思路是对的，而且也比“空做一个 todo-demo 上 k8s”更有说服力。

关键点是：你要把它包装成“实验环境和能力展示”，不是“把生产架构硬改成 k8s”。这两者在面试里差别很大。前者显得你有判断力，后者会显得你为了技术而技术。

我建议你这样做：
- 保持生产继续用 `docker compose`
- 新开 `k8s-lab` 分支，只做本地树莓派实验
- 优先用 `k3s`，不要一开始上完整 kubeadm 集群
- 先做单节点，再考虑多节点
- 先把现有服务映射到 k8s 基本对象，再谈进阶能力

你这个项目很适合拆成这些练习：
- `web-app` -> `Deployment + Service + PVC + probes`
- `articles-sync` -> 更适合做成 `CronJob`，而不是常驻服务
- `nginx-modsecurity` -> 第一阶段可以继续保留成一个 `Deployment + Service`
- 之后第二阶段再尝试改成 `Ingress + ingress-nginx`
- 配置项和 token -> `ConfigMap + Secret`
- SQLite 和渲染产物 -> `PersistentVolumeClaim`
- 镜像发布 -> 后续再补 `GHCR -> k8s rollout`

如果目标是找 DevOps/Cloud 相关工作，我建议你优先学这些，而不是一开始碰很重的东西：
- Pod / Deployment / Service / Ingress
- ConfigMap / Secret
- PVC / StorageClass
- Liveness / Readiness probes
- CronJob
- Rolling update / rollback
- `kubectl logs/describe/exec`
- Kustomize 或 Helm 二选一
- 一个最基本的 CI/CD 到 k8s 的流程

对求职最有价值的，不是“我会 k8s”这句话，而是你能清楚解释：
1. 为什么生产没上 k8s  
2. 为什么本地做了 k8s lab  
3. 你把现有 compose 架构怎么映射到 k8s  
4. 你遇到了哪些资源限制和设计权衡  
5. 你如何做 health check、持久化、配置管理和滚动发布  

这会比单纯背概念强很多。
