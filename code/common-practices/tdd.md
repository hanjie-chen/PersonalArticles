# Test-Driven Development

“红绿测试”（Red-Green-Refactor）是测试驱动开发（Test-Driven Development, TDD）中的核心循环。它将开发过程简化为三个不断重复的步骤，确保代码质量的同时保持开发的节奏感。

## 1. 什么是红绿测试？

红绿测试的过程通常被称为“TDD 循环”，主要分为以下三个阶段：

### 第一步：红（Red）

在编写任何业务代码之前，先写一个测试用例。

- 动作：运行测试。
- 结果：测试必然失败（报错或断言失败），因为你还没写实现逻辑。
- 意义：定义你想要实现的功能需求。

### 第二步：绿（Green）

编写最少的代码，让刚才失败的测试通过。

- 动作：快速实现功能，甚至可以使用“硬编码”等不完美手段。
- 结果：测试通过，进度条变绿。
- 意义：验证功能逻辑的正确性。

### 第三步：重构（Refactor）

在保持测试通过的前提下，优化刚才写的代码。

- 动作：消除重复代码、提高可读性、优化结构。
- 结果：测试依然为绿色。
- 意义：保证代码的长期可维护性。

## 2. 为什么要“先做失败的测试”？

很多人习惯先写代码再补测试，但 TDD 强调“测试先行”，尤其是“先看到失败”，主要有以下几个深度原因：

### 确保测试是有效的（防范假阳性）

如果你先写了功能代码，再写测试，而测试直接通过了，你无法确定这个测试是真的在检查你的逻辑，还是因为它本身写错了（例如漏掉了断言）。

- 先看到它失败，再看到它因为你的改动而成功，这证明了测试确实在监控该功能。

### 以“使用者”的视角设计 API

先写测试强迫你站在代码调用者的角度思考：这个函数叫什么名字好用？参数怎么传最顺手？

- 这避免了过度设计，确保你开发出的接口是实用且简洁的。

### 明确定义边界与需求

“失败的测试”其实就是一份自动化的规格说明书。在动手写业务逻辑前，你必须理清：

- 输入是什么？

- 预期输出是什么？

- 异常情况如何处理？

  当你写出那个失败的测试时，你已经把需求完全理清楚了。

### 提供重构的底气

如果你直接写代码，后期重构时你会担心“改这里会不会把别的地方搞坏”。

有了先行的测试，你可以大胆重构。只要测试依然是绿色的，就说明你的改动没有破坏原有的业务逻辑。

## 总结

“先红”是为了确定目标，“后绿”是为了达成目标，“重构”是为了优雅地走下去。 这种方式虽然在初期感觉速度较慢，但它能极大地减少后期的调试（Debug）时间，并防止代码演变成难以维护的“屎山”。
